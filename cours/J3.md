## Brancher l'app sur VOTRE base {toggle="true"}

Vous connaissez le rituel depuis J1 : on clone, on installe, on lance.

```bash
git clone <url-du-repo> le-spot
cd le-spot
npm install
```

Avant de lancer, il manque une chose : l'app ne sait pas à quelle base Supabase parler. C'est l'objet de l'exercice ci-dessous.

### Exercice — connectez votre propre Supabase

Vous avez modélisé Le Spot (MCD). Vous allez maintenant créer la vraie base et la brancher.

1. Créez un projet Supabase (vous savez faire).
2. Dans **SQL Editor**, exécutez le contenu de `supabase/migrations/0001_init.sql`, puis le bloc « 5 terrains » de `supabase/seed.sql`.
3. Copiez `.env.local.example` en `.env.local` et remplissez les 3 variables (Project Settings → API).
4. `npm run dev`, ouvrez http://localhost:3000.

<callout icon="🔍" color="gray_bg">
**Indice 1 — Où trouver les valeurs ?** Dans Supabase : Project Settings → API. `Project URL`, la clé `anon public`, et la clé `service_role` (secrète).
</callout>

<callout icon="🔍" color="gray_bg">
**Indice 2 — Comparez avec votre MCD.** Ouvrez `0001_init.sql`. Retrouvez vos entités (`courts`, `bookings`, `profiles`) et vos cardinalités sous forme de clés étrangères (`court_id`, `user_id`). Le `CHECK (...)` correspond à vos valeurs autorisées (indoor/outdoor…).
</callout>

<callout icon="🔍" color="gray_bg">
**Indice 3 — Ça plante au lancement ?** 9 fois sur 10 c'est le `.env.local` : mauvaise URL, clé tronquée, ou fichier nommé `.env` au lieu de `.env.local`. Relancez `npm run dev` après chaque modif du fichier.
</callout>

## La carte du repo : l'arborescence EST le routeur {toggle="true"}

Dans Next.js (App Router), **les dossiers de `src/app/` sont les URLs**. Pas de fichier de routes à configurer : la structure de dossiers fait le routage.

| Fichier | URL |
|---|---|
| `src/app/page.tsx` | `/` |
| `src/app/login/page.tsx` | `/login` |
| `src/app/mes-reservations/page.tsx` | `/mes-reservations` |
| `src/app/admin/page.tsx` | `/admin` |
| `src/app/terrains/[id]/page.tsx` | `/terrains/1`, `/terrains/2`… |

<callout icon="💡" color="blue_bg">
`[id]` entre crochets = **segment dynamique**. Le dossier `terrains/[id]` répond à n'importe quel id. La valeur arrive dans le code via `params`.
</callout>

### À vous : prédire avant de cliquer

Ouvrez `src/app/`. **Avant** d'ouvrir quoi que ce soit : quel fichier répond à `/mes-reservations` ? à `/terrains/3` ? Vérifiez. `layout.tsx` enveloppe toutes les pages (la barre de navigation y vit : `src/components/site-nav.tsx`).

## Le concept-pivot : où tourne votre code ? {toggle="true"}

Par défaut, dans l'App Router, **un composant est un Server Component** : il s'exécute sur le serveur, jamais dans le navigateur. Il peut donc parler à la base directement et lire des secrets — le navigateur ne voit que le HTML résultant.

Pour rendre un composant **interactif** (clics, état, `useState`), il faut le marquer `'use client'` en première ligne : il devient un **Client Component**, envoyé et exécuté dans le navigateur.

<callout icon="💡" color="blue_bg">
Règle de l'app : **serveur par défaut, client en îlots**. On ne passe en `'use client'` que là où l'interactivité l'exige. Cherchez la directive `'use client'` dans le repo — elle est rare, et c'est voulu.
</callout>

### Prédisez : serveur ou client ?

Pour chacun, devinez avant d'ouvrir le fichier (cherchez la directive en haut) :

| Fichier | Serveur ou client ? |
|---|---|
| `src/app/page.tsx` | ? |
| `src/components/day-view.tsx` | ? |
| `src/components/slot-button.tsx` | ? |
| `src/components/filters.tsx` | ? |

<callout icon="🔍" color="gray_bg">
**Réponse.** `page.tsx` et `day-view.tsx` = **serveur** (ils lisent les données, aucun `'use client'`). `slot-button.tsx` et `filters.tsx` = **client** (clic pour ouvrir la modale, `onChange` des filtres). Le planning est un Server Component qui ne contient que de petits îlots clients.
</callout>

### La frontière des secrets

`src/lib/supabase/client.ts` (navigateur) et `src/lib/supabase/server.ts` (serveur) utilisent tous deux `NEXT_PUBLIC_SUPABASE_ANON_KEY`. Le préfixe **`NEXT_PUBLIC_`** veut dire : « cette variable est envoyée au navigateur ». La clé `service_role`, elle, n'a pas ce préfixe → elle ne quitte jamais le serveur.

<callout icon="💡" color="blue_bg">
Ouvrez les DevTools (onglet Réseau) sur le planning : vous verrez la clé anon partir vers Supabase. C'est normal — elle est *censée* être protégée par la RLS. (On verra en J5-J6 ce qui se passe quand la RLS est absente…)
</callout>

## Lire un Server Component qui interroge Supabase {toggle="true"}

Ouvrez `src/app/page.tsx` (la vue du jour). Lisez-le de haut en bas, c'est le cœur de J3.

```tsx
const supabase = await createClient();              // client serveur
const { data: courts } = await supabase             // requête directe à la base
  .from("courts").select("*").eq("is_active", true).order("name");
```

Trois idées à repérer :

1. **`async` / `await`** : la fonction de page est asynchrone, elle attend la base avant de rendre le HTML.
2. **La requête est lue côté serveur**, puis les données descendent dans `<DayView>` via des **props**.
3. **L'état (jour, filtres) vient de l'URL** : `searchParams`. Changer de jour = changer l'URL (voir `src/components/date-nav.tsx`), pas un `useState`.

<callout icon="💡" color="blue_bg">
En Next.js 16, `searchParams` et `params` sont des **promesses** : on écrit `const params = await searchParams`. C'est nouveau — votre mémoire (et celle de l'IA) date peut-être d'avant.
</callout>

### Prédire → modifier

Sur le planning, ajoutez `?type=indoor` à l'URL. Que se passe-t-il ? Retrouvez dans `page.tsx` la ligne qui lit `type` et l'ajoute à la requête (`.eq("type", type)`). **Modifiez** l'ordre d'affichage : remplacez `.order("name")` par `.order("name", { ascending: false })`, rechargez.

## TypeScript en lecture {toggle="true"}

On ne vous demande pas d'écrire du TypeScript pointu, juste de le **lire**.

Ouvrez `src/lib/types.ts` :

```ts
export type CourtType = "indoor" | "outdoor";   // une "union" : l'une OU l'autre
export type Court = {
  id: number;
  name: string;
  type: CourtType;
  is_active: boolean;
};
```

<callout icon="💡" color="blue_bg">
Ces unions sont écrites **à la main**, en miroir des contraintes `CHECK` de la base (`check (type in ('indoor','outdoor'))`). Si vous ajoutez une valeur en base, il faut l'ajouter ici aussi. Le `type Court = {...}` décrit la forme d'une ligne : c'est ce que l'éditeur connaît quand vous tapez `court.`.
</callout>

### À vous

Dans `page.tsx`, survolez (dans VS Code) la variable `courtList`. Quel type affiche l'éditeur ? Retrouvez d'où il vient. Une union se lit « telle valeur **ou** telle valeur » ; un `?` ou `| null` se lit « peut être absent ».

## Tailwind + shadcn : où vivent styles et composants {toggle="true"}

Les classes comme `className="flex items-center gap-2"` sont du **Tailwind** : des utilitaires CSS posés directement dans le JSX. Pas de fichier `.css` à chercher.

Les composants `Button`, `Card`, `Dialog`… viennent de **shadcn/ui**. Particularité : ils ne sont pas cachés dans `node_modules`, ils sont **dans votre repo**, sous `src/components/ui/`.

<callout icon="💡" color="blue_bg">
Ouvrez `src/components/ui/button.tsx`. C'est VOTRE fichier : vous pouvez le lire et le modifier. shadcn ne livre pas une dépendance, il **copie le code chez vous**.
</callout>

### Prédire → casser → réparer

Dans `src/components/day-view.tsx`, un créneau libre est habillé de classes vertes (`bg-emerald-50`…). **Prédisez** ce que donne `bg-red-100`, faites le changement, rechargez. Puis remettez la valeur d'origine. (La friction — casser puis réparer — fait mieux comprendre qu'observer.)
