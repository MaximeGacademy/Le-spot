## Le flux d'écriture : une Server Action {toggle="true"}

Lire les données = Server Component. **Écrire** les données = **Server Action** : une fonction marquée `'use server'`, qui tourne sur le serveur, déclenchée par un bouton ou un formulaire.

<callout icon="💡" color="blue_bg">
Analogie no-code : une Server Action, c'est **un scénario Make / un workflow n8n** déclenché par un clic. Le bouton est le « trigger », la fonction serveur est le « scénario ».
</callout>

Ouvrez `src/app/actions/bookings.ts` :

```ts
"use server";
export async function createBooking(_prev, formData) {
  const supabase = await createClient();
  const { data: { user } } = await supabase.auth.getUser();   // qui réserve ?
  await supabase.from("bookings").insert({ court_id, user_id: user.id, date, start_hour });
  revalidatePath("/");                                        // rafraîchir le planning
}
```

Suivez le fil complet :

1. `src/components/day-view.tsx` (serveur) affiche les créneaux. Un créneau libre est un `<SlotButton>`.
2. `src/components/slot-button.tsx` (**client**, `'use client'`) : le clic ouvre une modale (`Dialog`).
3. Le formulaire de la modale a `action={handleSubmit}` qui appelle la **Server Action** `createBooking`.
4. La Server Action insère en base, puis `revalidatePath("/")` met le planning à jour.

### Prédire → observer

Réservez un créneau (connecté en `client@lespot.test` / `motdepasse`). **Avant** : où apparaît la nouvelle réservation ? Réponse : sur le planning (créneau « Pris ») et dans `/mes-reservations`. Retrouvez dans `bookings.ts` les deux lignes `revalidatePath(...)` qui provoquent ces rafraîchissements.

<callout icon="🔍" color="gray_bg">
**La porte d'authentification.** Déconnecté, cliquez sur un créneau : vous partez sur `/login`. Pourquoi ? Voir `slot-button.tsx` : `if (!isAuthenticated) { router.push("/login") }`. Le composant serveur `page.tsx` a transmis `isAuthenticated` en prop.
</callout>

## Exercice — ajouter un champ de bout en bout (les 3 couches) {toggle="true"}

Objectif : afficher pour chaque terrain s'il dispose d'un **éclairage nocturne**. Vous allez toucher **les 3 couches** : base → type → UI.

### Le travail

1. **Base** (Supabase, SQL Editor) : ajoutez une colonne booléenne à `courts`.
2. **Type** (`src/lib/types.ts`) : ajoutez le champ au `type Court`.
3. **UI** (`src/components/day-view.tsx`) : affichez un badge « 🌙 Nocturne » quand c'est vrai.

<callout icon="🔍" color="gray_bg">
**Indice 1 — Base.** `alter table courts add column has_lighting boolean not null default false;` Puis passez un ou deux terrains à `true` avec un `update`.
</callout>

<callout icon="🔍" color="gray_bg">
**Indice 2 — Type.** Dans `type Court`, ajoutez `has_lighting: boolean;`. Comme `page.tsx` fait `select("*")`, la donnée arrive déjà — il manque juste sa description côté TypeScript.
</callout>

<callout icon="🔍" color="gray_bg">
**Indice 3 — UI.** Dans `day-view.tsx`, à côté des `<Badge>` existants : `{court.has_lighting && <Badge>🌙 Nocturne</Badge>}`.
</callout>

<callout icon="💡" color="blue_bg">
La leçon : une feature « simple » traverse presque toujours **plusieurs couches**. Savoir où chacune vit (base / types / UI) = savoir lire et modifier n'importe quel repo.
</callout>

## Auth et route protégée {toggle="true"}

Ouvrez `src/app/mes-reservations/page.tsx`. C'est un Server Component qui commence par :

```tsx
const { data: { user } } = await supabase.auth.getUser();
if (!user) { redirect("/login"); }
```

<callout icon="💡" color="blue_bg">
La **session** est portée par des cookies, rafraîchis à chaque requête par `src/proxy.ts`. ⚠️ En Next.js 16, l'ancien `middleware.ts` s'appelle désormais **`proxy.ts`** — même rôle, nouveau nom. C'est lui qui maintient l'utilisateur connecté entre les pages.
</callout>

### À vous

Déconnecté, tapez l'URL `/mes-reservations` à la main. Que se passe-t-il ? (Redirection vers `/login`.) Retrouvez la ligne `redirect("/login")` responsable. On prépare ainsi J5-J6 : *cette* garde suffit-elle à tout protéger ? (Indice pour plus tard : une page protégée ≠ une donnée protégée.)

## Admin : le pattern CRUD {toggle="true"}

Ouvrez `src/app/admin/page.tsx`. C'est le **CRUD** complet des terrains : **liste → formulaire → suppression**.

- **Read** : la liste des terrains (`select("*").order("id")`).
- **Create** : le formulaire « Ajouter un terrain » → Server Action `createCourt`.
- **Update** : chaque terrain a un formulaire inline → `updateCourt`.
- **Delete** : un bouton → `deleteCourt`.

Toutes ces actions vivent dans `src/app/actions/courts.ts`.

### Prédire → observer

Connecté en `admin@lespot.test`, le lien **Admin** apparaît dans la barre du haut. Connecté en client, il disparaît. Retrouvez dans `src/components/site-nav.tsx` la condition `role === "admin"` qui masque le lien.

<callout icon="🔍" color="gray_bg">
**Question qui prépare J5-J6.** Le lien est masqué pour les clients, et la page `/admin` les renvoie. Mais l'**action** `deleteCourt`, elle, vérifie-t-elle le rôle ? Allez voir `courts.ts`… (On s'arrête là aujourd'hui : on observe, on ne corrige pas.)
</callout>

## Juger l'IA : repérer les anti-patterns {toggle="true"}

Le but final de ces deux jours : **savoir juger** le code qu'une IA vous pond. Quelques réflexes :

<callout icon="💡" color="blue_bg">
**1. Fetch en `useEffect` côté client.** Si vous voyez un composant `'use client'` qui fait `useEffect(() => { fetch(...) })` pour charger des données d'affichage → drapeau rouge. Ici on lit les données dans un **Server Component** (`page.tsx`). C'est plus simple, plus rapide, et le secret reste au serveur.
</callout>

<callout icon="💡" color="blue_bg">
**2. Tout en client.** Un gros `'use client'` en haut d'une page entière = JavaScript inutile envoyé au navigateur et secrets exposés. Le bon réflexe : serveur par défaut, `'use client'` en petits îlots (comme `slot-button.tsx`).
</callout>

<callout icon="💡" color="blue_bg">
**3. Secret exposé.** Une clé sensible préfixée `NEXT_PUBLIC_`, ou une clé `service_role` utilisée dans un Client Component → fuite. Vérifiez toujours **où tourne** le code qui touche un secret.
</callout>

### Exercice d'œil

Parcourez le repo et listez **où** se trouve chaque `'use client'`. Sont-ils tous justifiés (interactivité réelle) ? Pour chacun, demandez-vous : « pourrais-je remonter cette logique côté serveur ? »
